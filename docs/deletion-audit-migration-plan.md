# Critical deletion audit migration plan

Status: proposed; no schema or production function changes have been applied.
Explicit migration approval is required before implementation.

## Scope and invariant

Audit successful hard deletions from:

- `fines`
- `expenses`
- `debts`
- `debt_payments`

The live rows continue to be physically deleted, so every existing load,
financial filter, sum, and report keeps its current behavior. Historical rows
and historical financial values are not updated. Sales and attendance are out
of scope for this migration.

Soft-delete is not recommended for this stage because all readers would need a
new `deleted_at is null` condition; missing one condition could change totals.

## Proposed migration

Create `public.deletion_audit_log` with:

- `id bigint generated by default as identity primary key`
- `entity_type text not null` constrained to the four allowed table names
- `entity_id bigint not null`
- `deleted_row jsonb not null` containing the row immediately before deletion
- `deleted_by bigint not null references public.app_users(id)`
- `deleted_by_role text not null` copied from `app_users.role` at deletion time
- `deleted_at timestamptz not null default now()`

Add indexes on `(entity_type, entity_id)` and `deleted_at desc`. Enable RLS and
create no `anon` or `authenticated` policies. The frontend never loads this
table.

Create one `public.maestro_delete_with_audit(p_entity_type text,
p_entity_id bigint, p_actor_user_id bigint)` database function. It will:

1. Validate the entity type against a fixed `case` statement; no dynamic table
   names or client-supplied SQL.
2. Resolve and snapshot the actor role from `public.app_users`.
3. Select and lock the target row, insert its complete JSONB snapshot into the
   audit table, and delete the target row in one database transaction.
4. Preserve the existing seven-day fine deletion rule and current not-found
   response behavior.
5. Return only `{ ok, deleted, error? }`; the deleted snapshot never leaves the
   database.

The function should remain `security invoker`, be callable only by
`service_role`, and use explicit privileges:

- revoke execute from `public`, `anon`, and `authenticated`
- grant execute to `service_role`

## Edge Function change after migration approval

Update only the `delFine`, `delExpense`, `delDebtPayment`, and `delDebt` action
blocks in `supabase/functions/api/index.ts`. After the existing authorization
checks, call the RPC with `appUserResult.data.id`. Do not change action names,
request payloads, frontend calls, calculation code, or response success shape.

Before deployment, compare local `api` with remote version 27 again. Deploy the
migration first, then the Edge Function, so no deployed action can call a
missing RPC.

## Verification plan

1. Apply and test in the isolated Supabase test project first.
2. Verify `anon` and `authenticated` cannot select the audit table or execute
   the RPC; verify `service_role` can execute it.
3. In a transaction or disposable test fixture, delete one row of each type and
   assert exactly one audit row with the correct actor, role, timestamp, entity
   id, and complete snapshot.
4. Force an audit insert failure and assert the source row is not deleted.
5. Verify the expired-fine path still returns 403 and creates no audit row.
6. Run security and performance advisors.
7. Apply production migration, deploy `api`, verify the remote function version,
   and perform an owner-approved end-to-end deletion check using a newly created
   disposable record only.
8. Compare financial totals before and after the disposable create/delete pair;
   existing records and totals must be unchanged.

## Rollback

Revert the four API action blocks to direct deletes if the RPC causes failures.
Keep `deletion_audit_log` and its existing entries intact for forensic history.
Dropping the audit table or deleting audit rows requires a separate explicit
approval.
